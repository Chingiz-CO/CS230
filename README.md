# CS 230 Portfolio Reflection

## Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?

The client for this project was **The Gaming Room**, a company that wanted to expand its game, *Draw It or Lose It*, from an Android-only application into a web-based game that could support multiple operating platforms. The software needed to allow multiple teams and players, ensure that game and team names remained unique, support only one active game instance at a time, and provide a scalable and secure architecture that could accommodate future growth.

## What did you do particularly well in developing this documentation?

One of my strengths was analyzing the client's requirements and comparing different operating platforms before making a recommendation. I also organized the design document clearly, explaining how the chosen architecture would satisfy both the technical and business requirements. This helped communicate the proposed solution in a logical and professional manner.

## What about the process of working through a design document did you find helpful when developing the code?

Creating the design document before writing code helped me understand the project as a whole. It encouraged me to think about system architecture, software requirements, platform compatibility, and potential challenges before implementation. Having a clear design reduced uncertainty and provided a roadmap that would make development more organized and efficient.

## If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could revise one section, I would expand the discussion of security and cloud deployment. I would include more detailed recommendations for authentication, data protection, scalability, and deployment strategies to better prepare the application for production environments.

## How did you interpret the user's needs and implement them into your software design? Why is it so important to consider the user's needs when designing?

I interpreted the user's needs by carefully reviewing the project requirements and identifying the features that would provide the greatest value. These requirements guided decisions about the software architecture, operating platform, scalability, and overall system design. Considering the user's needs is essential because successful software should solve real problems while remaining reliable, easy to use, and adaptable to future requirements.

## How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?

I approached software design by first understanding the client's goals, then evaluating multiple technical solutions before selecting the most appropriate architecture. I divided the project into manageable components and considered factors such as maintainability, scalability, security, and performance throughout the design process. In future projects, I will continue using structured design documents, requirements analysis, architectural planning, and iterative refinement to develop software that meets both technical and business objectives.
